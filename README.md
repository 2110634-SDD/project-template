### **รายละเอียดโครงงาน (Project Details) - วิชา 2110634 Software Design and Development**

#### **1. ภาพรวมและขอบเขตของโครงงาน (Project Scope)**

โครงงานนี้มีวัตถุประสงค์เพื่อให้นิสิตได้ประยุกต์ใช้หลักการและแนวคิดทั้งหมดที่ได้เรียนในวิชา Software Design and Development ตั้งแต่การวิเคราะห์ความต้องการ, การออกแบบสถาปัตยกรรม, การออกแบบรายละเอียด, ไปจนถึงการลงมือพัฒนาและปรับปรุงคุณภาพของซอฟต์แวร์ ผ่านกระบวนการพัฒนาแบบ Agile ที่มีการทำงานเป็นรอบ (Iterations) และมีการส่งมอบงานเป็นส่วนย่อย (Milestones)

**ขอบเขตทางเทคนิค (Technical Scope):**
ทุกกลุ่มจะต้องพัฒนาซอฟต์แวร์ที่ประกอบด้วย 2 ส่วนหลักที่ทำงานร่วมกัน:
1.  **Backend Application:** ระบบฝั่งเซิร์ฟเวอร์ที่ทำหน้าที่เป็น Business Logic และจัดการข้อมูล โดยให้บริการผ่าน RESTful APIs
2.  **Mobile Application:** แอปพลิเคชันบนมือถือ (สามารถเลือกใช้เทคโนโลยีใดก็ได้ เช่น Native Android/iOS, React Native, Flutter) ที่ติดต่อกับ Backend เพื่อให้บริการแก่ผู้ใช้

#### **2. ธีมของโครงงาน (Project Theme)**

เพื่อให้สามารถประเมินคุณภาพของการออกแบบได้อย่างยุติธรรมและเป็นมาตรฐานเดียวกัน ทุกกลุ่มจะต้องเลือกพัฒนาโครงงานภายใต้ **"โดเมนของปัญหา (Problem Domain)"** ที่กำหนดให้ โดยมีอิสระในการสร้างสรรค์รายละเอียดและฟีเจอร์เฉพาะของกลุ่มตนเอง

**ตัวอย่างธีมของโครงงาน (Project Theme Examples):**
*   **ธีม A: ระบบจัดการการจอง (Booking/Reservation Management System)**
    *   *ตัวอย่างโครงงาน:* ระบบจองคิวร้านอาหาร, ระบบจองห้องประชุม, ระบบจอง Co-working space, ระบบนัดหมายแพทย์
    *   *ความท้าทายเชิงออกแบบ:* การจัดการสถานะ (State), การป้องกันการจองซ้อน (Concurrency), การออกแบบโมเดล Resource และ Timeslot
*   **ธีม B: ระบบ E-commerce ที่มีโปรโมชันซับซ้อน (E-commerce with Complex Promotions)**
    *   *ตัวอย่างโครงงาน:* ร้านขายหนังสือออนไลน์พร้อมโปรโมชัน, ร้านขายอุปกรณ์คอมพิวเตอร์แบบจัดสเปค, แพลตฟอร์มขายคอร์สเรียนแบบ Subscription
    *   *ความท้าทายเชิงออกแบบ:* การประยุกต์ใช้ Strategy Pattern หรือ Decorator Pattern ในการจัดการส่วนลด, การออกแบบระบบตะกร้าสินค้า (Shopping Cart), การจัดการสต็อกสินค้า

### **3. รายละเอียดและเกณฑ์การประเมินในแต่ละ Milestone**

โครงงานจะแบ่งการทำงานและประเมินผลออกเป็น 3 Milestones หลักดังนี้:

#### **Milestone 1: Foundations & High-Level Design (ส่งสัปดาห์ที่ 7)**
*   **เป้าหมาย:** "เราจะสร้างอะไร และจะสร้างมันอย่างไรในภาพรวม?" เน้นที่ความเข้าใจในปัญหาและกำหนดทิศทางการออกแบบ
*   **สิ่งที่ต้องนำเสนอ (Presentation Focus):**
    *   นำเสนอ Problem Domain และขอบเขตของโครงงานที่กลุ่มเลือก
    *   อธิบาย Use Cases หลัก และลงรายละเอียดใน Main Flow และ Alternative Flows ของ Use Case ที่สำคัญที่สุด 1-2 Use Cases
    *   นำเสนอ Conceptual Model (Class Diagram) และให้เหตุผลเบื้องหลังการออกแบบ
    *   อธิบาย Quality Attributes ที่สำคัญของระบบและสถาปัตยกรรมเบื้องต้นที่เลือกใช้
*   **สิ่งที่ต้องส่งมอบ (Deliverables):**
    1.  **เอกสารในโฟลเดอร์ `docs/milestone-1/`:**
        *   `use-case-descriptions.md`: เอกสาร Use-Case Description แบบ "Fully Dressed"
        *   `conceptual-model.md`: Conceptual Class Diagram พร้อมคำอธิบาย
    2.  **ติดป้าย (Tag)** คอมมิตสุดท้ายของการส่งงานใน Git ว่า `v1.0-M1`

| เกณฑ์การประเมิน Milestone 1 (10% ของคะแนนรวม) | น้ำหนัก |
| :--- | :---: |
| 1. คุณภาพของ Use-Case Modeling | 40% |
| 2. คุณภาพของ Conceptual Design และการระบุ QAs | 30% |
| 3. การนำเสนอและการตอบคำถาม | 30% |

#### **Milestone 2: Architectural Checkpoint - "The Walking Skeleton" (Demo สัปดาห์ที่ 13)**
*   **เป้าหมาย:** "พิสูจน์ว่าสถาปัตยกรรมที่ออกแบบไว้ทำงานได้จริง" ผ่านการพัฒนาฟังก์ชันหลัก 1 ฟังก์ชันให้ครบวงจร
*   **สิ่งที่ต้องนำเสนอ (Presentation Focus):**
    *   **Live Demo:** สาธิตการทำงานของ 1 Use Case ที่เลือกมาให้ทำงานได้ครบ End-to-End (จาก Mobile App -> Backend -> Database -> Mobile App)
    *   **CI/CD Demo:** แสดงผลการทำงานของ CI Pipeline (เช่น เครื่องหมายถูกสีเขียว) ที่ทำการ Build และ Test โค้ดโดยอัตโนมัติ
    *   **Code Walkthrough:** อธิบายโค้ดในส่วนที่สะท้อนการตัดสินใจเชิงออกแบบที่สำคัญ (เช่น การใช้ Design Pattern, การแยก Layer)
*   **สิ่งที่ต้องส่งมอบ (Deliverables):**
    1.  **Live Demo** ในชั้นเรียน
    2.  **Source Code** ที่มี CI/CD Pipeline (`.gitlab-ci.yml` หรือไฟล์ใน `.github/workflows/`)
    3.  **เอกสารในโฟลเดอร์ `docs/milestone-2/`:**
        *   `detailed-design.md`: Sequence Diagram และ Refined Class Diagram สำหรับ Use Case ที่ทำ Demo
    4.  **ติดป้าย (Tag)** คอมมิตสุดท้ายของการส่งงานใน Git ว่า `v2.0-M2`

| เกณฑ์การประเมิน Milestone 2 (15% ของคะแนนรวม) | น้ำหนัก |
| :--- | :---: |
| 1. การทำงานและความสมบูรณ์ของ "Walking Skeleton" | 40% |
| 2. คุณภาพของการออกแบบรายละเอียดและโค้ดเบื้องต้น | 40% |
| 3. การสาธิตและการอธิบายเชิงเทคนิค | 20% |

#### **Milestone 3: Final Product & Project Portfolio (ส่งสัปดาห์ที่ 16)**
*   **เป้าหมาย:** "ส่งมอบผลิตภัณฑ์ที่ทำงานได้สมบูรณ์และมีคุณภาพ พร้อมทั้งแฟ้มผลงานที่สะท้อนกระบวนการเรียนรู้"
*   **สิ่งที่ต้องนำเสนอ (Presentation Focus):**
    *   **Final Demo:** สาธิตการทำงานของฟังก์ชันหลักทั้งหมดของแอปพลิเคชัน
    *   **สรุปการออกแบบ:** นำเสนอภาพรวมสถาปัตยกรรมสุดท้าย และการตัดสินใจเชิงออกแบบที่สำคัญที่เกิดขึ้นระหว่างทาง
    *   **บทเรียนและความท้าทาย:** สรุปปัญหาที่พบ, แนวทางการแก้ไข, และสิ่งที่ทีมได้เรียนรู้จากโครงงานนี้
*   **สิ่งที่ต้องส่งมอบ (Deliverables):**
    1.  **Final Presentation** ในชั้นเรียน
    2.  **Project Portfolio (แฟ้มผลงานของโครงงาน) ทั้งหมดใน Git Repository:**
        *   Source Code ฉบับสมบูรณ์ของทั้ง `backend/` และ `mobile/`
        *   โฟลเดอร์ `docs/` ที่มีเอกสารการออกแบบทั้งหมด (ที่ผ่านการปรับปรุงจากฟีดแบค)
        *   ไฟล์ `README.md` ที่สมบูรณ์ (มีวิธี Setup และ Run โครงงาน)
    3.  **ติดป้าย (Tag)** คอมมิตสุดท้ายของการส่งงานใน Git ว่า `v3.0-Final`

| เกณฑ์การประเมิน Milestone 3 (15% ของคะแนนรวม) | น้ำหนัก |
| :--- | :---: |
| 1. ความสมบูรณ์และเสถียรภาพของฟังก์ชันโดยรวม | 30% |
| 2. คุณภาพของการออกแบบและโค้ดฉบับสมบูรณ์ (Clean Code, Refactoring) | 50% |
| 3. การนำเสนอผลงานและสรุปผลการเรียนรู้ | 20% |

### **4. สิ่งที่ต้องทำใน GitHub Classroom**

วิชานี้จะใช้ GitHub Classroom เป็นเครื่องมือหลักในการจัดการและส่งมอบโครงงาน ซึ่งจะช่วยสร้าง "Project Portfolio" ของแต่ละกลุ่มโดยอัตโนมัติ

**ขั้นตอนการทำงาน:**
1.  **การสร้าง Project Repository:**
    *   อาจารย์จะสร้าง "Group Assignment" ชื่อ "Project" ใน GitHub Classroom
    *   ตัวแทนของแต่ละกลุ่มจะต้องกด "Accept" Assignment เพื่อให้ระบบ **สร้าง Private Repository สำหรับกลุ่มโดยอัตโนมัติ**
    *   Repository นี้จะมาพร้อมกับโครงสร้างโฟลเดอร์เริ่มต้น (`backend/`, `mobile/`, `docs/`) และไฟล์ที่จำเป็น (`README.md`, `.gitignore`)

2.  **การทำงานในแต่ละวัน:**
    *   ให้นิสิตทำงานใน **Branch** ของตัวเอง และสร้าง **Pull Request (PR)** เมื่อต้องการรวมโค้ดเข้ากับ Branch หลักของทีม เพื่อให้เกิดการ Review โค้ดกันภายในกลุ่ม
    *   **เอกสารทั้งหมดให้ทำในรูปแบบไฟล์ Markdown (`.md`)** และเก็บไว้ในโฟลเดอร์ `docs/` การแก้ไขเอกสารก็ให้ทำผ่านการ `commit` เหมือนกับการแก้ไขโค้ด

3.  **การส่งงานในแต่ละ Milestone:**
    *   เมื่อถึงกำหนดส่งงานของแต่ละ Milestone ให้นิสิต **ไม่ต้องสร้างเอกสารหรือ zip ไฟล์ใหม่เพื่อส่ง**
    *   วิธีการส่งคือการ **"ติดป้าย (Tag)"** บนคอมมิตสุดท้ายของงานใน Milestone นั้นๆ ด้วยคำสั่ง Git:
        *   **Milestone 1:** `git tag -a v1.0-M1 -m "Submission for Milestone 1"`
        *   **Milestone 2:** `git tag -a v2.0-M2 -m "Submission for Milestone 2"`
        *   **Milestone 3:** `git tag -a v3.0-Final -m "Final Submission"`
    *   หลังจาก Tag แล้ว ให้ทำการ `git push --tags` เพื่อส่ง Tag ขึ้นไปยัง GitHub Repository
    *   **อาจารย์จะตรวจงานโดยยึดตามโค้ดและเอกสาร ณ จุดที่ Tag ไว้** และสามารถเปรียบเทียบความเปลี่ยนแปลงระหว่าง Tag ได้

4.  **การตั้งค่า Auto-Testing (CI):**
    *   ใน Milestone 2 นิสิตจะต้องสร้างโฟลเดอร์ `.github/workflows/` และสร้างไฟล์ `ci-pipeline.yml` เพื่อตั้งค่า GitHub Actions
    *   เป้าหมายคือ ทุกครั้งที่มีการ Push โค้ดขึ้นไป ระบบจะทำการ Build และ Run Unit Tests โดยอัตโนมัติ และแสดงผลเป็นเครื่องหมาย ถูกสีเขียว/ผิดสีแดง ใน Repository
